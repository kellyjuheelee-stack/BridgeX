# BridgeX 남은 기능 마스터 로드맵 (Vercel + Supabase)

- 작성일: 2026-07-11
- 작성자 역할: **기획자 세션** (스펙/플랜만, 코드 금지 — [[role-planner-not-developer]])
- 대상 스택: Next.js(App Router, TS) on Vercel + Supabase(Postgres/Auth/Storage). **정적HTML+Express 구 스택은 폐기 대상 — 신규 기획은 `web/`만 대상.**
- 목적: 마이그레이션(P1~P5) 이후 남은 작업을 **전환 퍼널 기여도 × 착수 준비도**로 우선순위화하고, 개발자 세션이 바로 집어갈 수 있게 에픽 단위로 정리한다.

---

## 0. 전제 — 현재 확정된 것 (충돌 방지선)

개발자 세션이 이미 구현·main 병합한 것들. **아래와 상충되는 재설계는 하지 않는다.**

- **데이터 모델**: `public.profiles`(id, name, company_name, phone, is_admin, roadmap_progress jsonb …), `public.export_diagnosis_requests`(진단·컨설팅·미팅 전부 한 테이블, `consulting_stage`/`consulting_checklist jsonb`/`meetings jsonb`/`admin_memo`/`member_id`/`product_files jsonb`/`checklist_answers jsonb` …). 마이그레이션 `0001_init.sql`, `0002_diagnosis_checklist.sql`.
- **인증/권한**: Supabase Auth + `profiles.is_admin`. 쓰기·관리자 조회는 **서버(서비스 롤)** 가 수행, RLS는 2차 방어(회원 self-select만 허용). 순수함수 `resolveAccess` + `requireAdmin` 게이트.
- **라우트**: `/` `/login` `/signup` `/diagnose` `/diagnose/result/[id]` `/mypage` `/tools/{email,catalog,reply}` `/admin` `/admin/leads/[id]` `/privacy` `/terms`.
- **엔진(순수함수, 규칙기반, Claude 교체지점 격리)**: `generateDiagnosis`, `toRow`, `buildRoadmap`+`fromRow`, `email/templates`, `catalog/audit`, `reply/replyAssistant`, `consulting` 상수. vitest 47개 통과.
- **불변 원칙** (기획이 반드시 계승):
  - **전환 원칙 A**: 무료 훅은 규칙기반(저비용, 가치 증명 + "혼자 못 함" 벽 노출) → 매출은 유료회원·전문가 컨설팅.
  - **법적 안전**: 카탈로그/규제 도구는 "금지/위반/불법" **단정 금지**, 판단은 전문가로 이관(테스트가 강제).
  - **AI 바이어 자동발굴 금지**: 할루시네이션 위험 → **실데이터 + 사람(대표님) 검증만.**
  - **발송 비대행**: 이메일/답장 도구는 회원이 복사해 직접 발송, BridgeX가 대신 보내지 않음.

### 마이그레이션에서 넘어온 알려진 갭 (미완결 항목)
1. 상담 CTA의 `stepContext`(어느 로드맵 단계에서 상담 신청했는지)가 새 스키마에 **미저장** — 구 스택은 `admin_memo`에 기록했음.
2. 로드맵 catalog 분기: `product_files` 미수집 → `hasBrochure` 항상 false → `catalog_audit` 단계가 안 뜸.
3. 관리자 목록: 필터/정렬/페이지네이션/임시진단 버튼 미포팅(전건 최신순 나열만).
4. `product_files` 첨부의 업로드·서빙(Storage) 스토리 없음.

## 1. 우선순위 백로그 (에픽 단위)

각 에픽: **목적 → 전환 기여 → 범위 → 데이터 변경 → 충돌 여부 → 규모(S/M/L)**.
권장 착수 순서 = 아래 P0 → P3 순.

### P0. 런칭 필수 (기획 대상 아님, 사용자+개발자가 라이브 Supabase로 실행)
- Supabase 프로젝트 생성 → `0001`→`0002` SQL 실행 → Email confirm 임시 off → 키 3개 `.env.local` → `smoke:auth`·`smoke:diagnose` 검증 → 관리자 `is_admin=true` → Vercel 배포(Root=`web`). 런북 `web/README.md`.
- 약관/방침 placeholder(사업자명·보호책임자·문의 이메일·위탁내역)를 실제값으로 채우고 법률 검토.
- → **이건 스펙 대상이 아니라 배포 체크리스트.** 나머지 기획은 이 위에서 동작한다고 가정.

---

### P1. 관리자 백오피스 완성 — 리드 운영·전환 엔진 【권장 1순위】
- **목적**: 진단으로 들어온 리드를 컨설턴트(대표님)가 실제로 전환시키는 운영 화면 완성. 갭 ①③⑥ 통합.
- **전환 기여**: ★★★ (리드→상담→컨설팅 전환의 실무 병목 해소. 매출에 가장 직접적.)
- **범위**:
  - 상담 신청 `stepContext` 저장·표시 (갭①). 리드 상세에서 "어느 단계 벽에서 상담을 신청했는지" 노출 → 상담 준비 컨텍스트.
  - 관리자 목록 강화 (갭③): 상태/컨설팅단계/카테고리 **필터**, 점수·최신순 **정렬**, **페이지네이션**, 핫리드(`consultation_requested`) 상단 고정.
  - 관리자에서 **회원 로드맵 진행 보기** (다음 후보): 리드 상세에 해당 회원의 `roadmap_progress`(현재 단계 N/16, 완료율, 막힌 🔴단계)를 읽기전용 표시 → 대화 준비.
- **데이터 변경**: `export_diagnosis_requests`에 `consultation_step_context text` 추가(경량). 나머지는 조회 로직만.
- **충돌 여부**: 없음. 기존 `/admin`·`/admin/leads/[id]`·`consulting` 상수 **확장**만. 데이터 모델 계승.
- **규모**: M. → **상세 스펙: `2026-07-11-admin-backoffice-completion-design.md` (본 문서와 함께 산출).**

### P2. 제품/카탈로그 첨부 파일 (Supabase Storage) + 로드맵 catalog 분기 활성화
- **목적**: 진단·카탈로그 도구에서 회원이 브로셔/제품자료를 **업로드**하고, 관리자가 열람. 갭 ②④ 해소.
- **전환 기여**: ★★ (브로셔 유무 분기 → 카탈로그 감사/생성 훅 활성화. 콘텐츠형 전환의 전제.)
- **범위**: 진단 폼·마이페이지에 파일 업로드(브로셔·제품 이미지·성분표) → Storage 버킷 저장, `product_files jsonb`에 메타 기록. 관리자 리드 상세에서 **서명 URL**로 열람. `buildRoadmap`의 `hasBrochure`가 실제 업로드 여부로 결정되도록 `toRow`/`fromRow` 매핑 연결.
- **데이터 변경**: Storage private 버킷 1개 + RLS(회원 self / 서비스롤 all). `product_files` 스키마 확정(파일명·경로·타입·업로드시각).
- **충돌 여부**: 없음. `product_files` 컬럼은 이미 존재 → 채우는 스토리만 추가.
- **규모**: M.

### P3. 바이어 데이터베이스 + 리드↔바이어 수동 매칭 (관리자 전용) 【대표님 코어 자산】
- **목적**: 전문가 컨설팅의 실제 상품인 **바이어 매칭**을 시스템화. 대표님이 보유한 바이어 리스트를 백엔드에 안전 보관 + 리드에 **사람이 판단해** 매칭.
- **전환 기여**: ★★★ (유료 컨설팅의 딜리버리 자산. 단, 무료 훅이 아니라 유료 단계.)
- **범위**:
  - `buyers` 테이블 + 관리자 CRUD 화면(`/admin/buyers`). 국가·채널·카테고리·상태 태그, 비공개 메모.
  - 리드 상세에서 **관리자가 수동으로** 후보 바이어 연결(매칭 상태: 후보/제안됨/미팅/계약). **AI 자동발굴 금지 원칙 준수.**
  - 회원 마이페이지엔 바이어 실명 미노출 — "매칭 진행 상태"만(원칙: 코어 자산 보호).
- **데이터 변경**: `buyers`, `lead_buyer_matches`(리드↔바이어 N:M) 신규 테이블 + RLS(관리자/서비스롤 전용, 회원 접근 불가).
- **충돌 여부**: 없음(신규 테이블). 기존 컨설팅 트랙과 연결만.
- **규모**: L. → 별도 상세 스펙 필요(P1 이후).

### P4. 알림/이메일 발송 인프라 (상담 신청 트리거)
- **목적**: 상담 신청·핫리드 발생 시 관리자에게 실제 알림. 구 스택은 콘솔 로그 모드였음([[pending-smtp-email]]).
- **전환 기여**: ★★ (리드 반응 속도 = 전환율. 놓치는 리드 방지.)
- **범위**: 상담 신청 서버액션에서 관리자 알림(수신 skigirl77@naver.com). Supabase/외부 발송(Resend 등) 중 택1 — **결정 필요 항목**(스펙에서 옵션 제시). 회원 대상 발송은 P0의 "대행 안 함" 원칙과 구분(트랜잭션 알림만 허용).
- **데이터 변경**: 없음(외부 서비스 env). 발송 로그 테이블은 옵션.
- **충돌 여부**: 없음.
- **규모**: S~M.

### P5. Claude API 연동 — 규칙기반 엔진의 personalize 교체
- **목적**: `generateDiagnosis`/`email/templates`/`catalog/audit`/`reply` 의 **격리된 교체지점**을 실제 LLM으로 승격.
- **전환 기여**: ★★ (품질↑, 유료 특전 차별화. 단 비용 발생 → 유료 게이팅과 함께.)
- **범위**: 교체지점만 LLM 호출로 교체, **규칙기반 폴백 유지**(할루시네이션·비용·장애 방어). 진단 점수 산정은 규칙 유지(설명 가능성), personalize/문구 생성만 LLM. 법적 안전 원칙(단정 금지) 프롬프트·후처리로 강제.
- **데이터 변경**: 없음. API 키 env. 사용량/비용 로깅 옵션.
- **충돌 여부**: 없음(교체지점이 이미 격리 설계됨).
- **규모**: M (도구별로 쪼갤 수 있음).

### P6. 유료 게이팅 / 결제 + 2b AI 카탈로그 생성기
- **목적**: 무료↔유료 경계 실체화. 2b 카탈로그 생성기(브로셔 없는 회원용 AI 생성, 유료 특전)는 결제가 전제.
- **전환 기여**: ★★★ (매출화의 최종 관문) — 단 **의존성 큼**(P2 Storage, P5 Claude, 결제 PG 선정).
- **범위**: 요금제 정의 → 게이팅 지점(로드맵 'soon' 단계·2b 생성기) → 결제 연동 or "문의/대기" 플래그(무결제 시). 2b 생성기 = 제품정보→AI 브로셔 초안(유료), 무료 회원엔 프로급 샘플 갤러리 티저.
- **데이터 변경**: `profiles.plan`/구독 상태, 결제 이력 테이블.
- **충돌 여부**: 없음(신규). 로드맵 'soon' 단계와 연결.
- **규모**: L. **가장 마지막**(선행 의존성 다수).

## 2. 권장 실행 순서와 근거

```
P0(배포·법무, 사용자)  →  P1(관리자 완성)  →  P2(Storage 첨부)  →  P3(바이어 DB)
                                                        ↘  P4(알림)  ↘  P5(Claude)  →  P6(결제·2b)
```

- **P1을 1순위로 두는 이유**: 신규 테이블·외부 의존 없이 기존 자산 확장만으로 "리드가 실제로 전환되는" 운영 루프를 완성한다. 갭 ①③⑥을 한 번에 닫고, 이후 모든 에픽(바이어 매칭·알림)이 붙을 관리자 화면 토대를 마련한다. 규모 M, 충돌 0.
- **P2를 2순위로**: 콘텐츠형 전환(카탈로그 감사/생성)의 물리적 전제. Storage 도입은 이후 2b 생성기까지 재사용.
- **P3(바이어)**: 유료 딜리버리의 핵심 자산이지만 규모 L·회원 미노출이라 운영툴(P1) 이후가 안전.
- **P4/P5는 병렬 가능**(서로 독립). P6는 선행 의존성 모두 충족 후 마지막.

## 부록. 명시적 가정 (묻지 않고 진행한 판단 — 사용자 수정 시 반영)
- A1. 신규 기획은 `web/`(Next.js/Supabase)만 대상, 구 정적/Express 스택은 손대지 않는다.
- A2. 결제 PG·LLM 벤더·발송 서비스 등 **외부 서비스 선정은 사용자 결정 사항** → 스펙에선 옵션만 제시, 확정은 보류.
- A3. 우선순위는 "전환 퍼널 기여 × 착수 준비도" 기준. 사용자가 특정 기능(예: 바이어 DB)을 먼저 원하면 순서 조정.
- A4. 모든 신규 데이터 변경은 **추가(add column / new table)** 로만 — 기존 컬럼 삭제·타입 변경 없음(개발자 구현물 보호).
