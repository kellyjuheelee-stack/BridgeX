# 수출 준비도 진단 개편 설계 — 체크리스트형 자가진단

- 날짜: 2026-07-09
- 상태: 승인됨 (스펙 확정, 플랜 작성 대기)
- 범위: STEP 2 수출 준비도 진단의 입력 UX 개편(입력폼 → 준비도 체크리스트)
- 대상 스택: **새 스택(Next.js + Supabase)** — 마이그레이션 **P2**의 진단 부분 상세 설계
- 관계: [Vercel + Supabase 마이그레이션 설계](2026-07-09-vercel-supabase-migration-design.md)의 P2를 이 방식으로 구현

## 배경 / 목적

현재 수출 준비도 진단(`diagnose.html`)은 **4단계 혼합 입력폼**이다. 자유입력·드롭다운 항목이 많아 고객의 작성 부담이 크고, 진입 단계에서 이탈이 발생한다.

이를 **"해당되는 것을 모두 체크하세요" 준비도 체크리스트**로 개편한다. 고객은 타이핑 대신 자사에 해당하는 항목을 체크하고, 그 결과로 진단 리포트를 받는다. 인지부하를 낮춰 완료율을 높이고, "체크 안 한 것 = 갭"이라는 직관적 대응으로 결과의 설득력도 높인다.

### 목표
1. 진단 문항을 체크박스 서술문(자가진단)으로 전환해 작성 부담을 낮춘다.
2. 리드(연락처) 확보는 그대로 유지한다 — 이 진단은 여전히 리드 생성 훅이다.
3. 기존 결과 구조(4영역 점수·우선과제·갭·EU현황)와 채점 기준을 그대로 계승한다.

### 비목표 (YAGNI)
- 채점 공식·가중치·임계값 변경 — 개편은 입력 UX이지 채점 기준이 아니다. 기존 값 계승.
- 익명(연락처 없이) 진단 — 리드 확보를 위해 연락처는 계속 수집한다.
- 마케팅 수신 동의·동의 이력 DB 저장 — 이번 범위 아님. 필수 동의 1줄만 유지.
- 결과 화면 재설계 — 기존 결과 화면을 그대로 계승.
- Claude API personalize — 별도 후속.

## 확정된 결정 (브레인스토밍 결과)

| 항목 | 결정 |
|---|---|
| 기존 진단과의 관계 | **교체/개편** (별도 신규 아님) |
| 진단 성격 | 리드폼 유지 + 진단 문항을 체크 UX로 |
| 선택 형태 | **준비도 체크리스트형**(서술문 체크박스, "해당되는 것 모두 체크") |
| 대상 스택 | 새 스택(Next.js) — 마이그레이션 P2에 반영 |
| 채점 | 기존 4영역 점수·우선과제·갭·EU현황 출력 및 공식/가중치 **그대로 계승** |
| 개인정보 동의 | 필수 동의 1줄 유지(기존대로), 제출 2지점 게이트 |

## 전체 구조 / 플로우

```
[1] 시작 정보 (최소 타이핑, 리드 확보용)
    · 연락처: 담당자명 · 회사명 · 이메일 · 전화        (필수, text)
    · 제품 식별: 대표 제품명(text) · 카테고리(택1) · 목표 국가(칩)
    · 홈페이지 / SNS (선택)

[2] 수출 준비도 체크리스트  ← 개편의 핵심
    "해당되는 것을 모두 체크하세요" — 4개 영역으로 그룹핑된 서술문 체크박스
    스크롤 한 판. 타이핑 없음. 안 체크하면 = 갭.

[3] 진단 결과 (기존 결과 화면 계승)
    · 종합점수 + 준비수준 배지
    · 4영역 점수 카드 + 영역별 부족 항목 칩
    · 우선 해결 과제 Top 3
    · EU 필수요건 현황 (보유 n/7 + 미비 목록)
    · 맞춤 요약 + 전문가 상담 전환 CTA (훅)
```

**유지되는 것**
- 기존 2개 유입: ① 회원가입→진단(기본정보 프리필) ② 비회원 진단→결과화면에서 비밀번호 설정해 가입(진단이 계정에 연결). 새 스택에선 ②의 가입이 Supabase Auth `signUp`.
- 결과 화면의 풍부한 출력과 상담 전환 훅.

**바뀌는 것**
- 진단 문항 입력 방식: 자유입력/드롭다운 → 체크박스 서술문.
- 채점 엔진의 **입력 구조**(출력 구조·공식은 계승).

> **마이그레이션 원칙의 예외 명시**: 마이그레이션 스펙은 "규칙 엔진은 로직 변경 없이 그대로 이식"이 원칙이나, **이 진단만은 의도적 개편**(입력 구조 변경)이다. 채점 공식·가중치·출력은 계승하되 입력 매핑은 새로 짠다.

## 준비도 체크리스트 문항

"우리 브랜드는 ~을 갖췄다" 형태. **체크 = 보유**, 미체크 = 갭. 각 영역 상단에 1줄 설명을 두어 모르는 항목도 학습되게 한다. "모르겠음/없음" 선택지는 두지 않는다(미체크가 곧 없음).

### 🟦 제품 경쟁력
- ☐ 국내에서 판매 중이다 (자사몰·스마트스토어·오프라인 등)
- ☐ 월 판매량 또는 베스트셀러 실적이 있다
- ☐ 제조 인증을 보유했다 (GMP·ISO 22716 등)

### 🟦 EU 규제 준비 (필수 7요건)
> 영역 설명: EU 화장품 판매에는 아래 7가지가 필수입니다.
- ☐ EU 책임자(RP)를 지정했다
- ☐ CPNP 사전 등록을 완료했다
- ☐ 제품정보파일(PIF)을 구비했다
- ☐ 제품안전성보고서(CPSR)를 작성했다
- ☐ 전성분(INCI) EU 규정 적합성을 확인했다
- ☐ EU 라벨링 요건을 충족했다
- ☐ 향료 알레르겐(80종) 표시에 대응했다

### 🟦 EU 포장 규제 (PPWR)
> 영역 설명: 2026년 8월 시행되는 EU 포장·포장폐기물 규정입니다.
- ☐ 재활용 가능 포장재를 사용한다
- ☐ 적합성 선언서(DoC)를 구비했다
- ☐ 포장 기술 문서를 구비했다
- ☐ 재생원료 함량을 확인했다

### 🟦 영업 자료
- ☐ 바이어에게 보낼 영문 카탈로그·회사소개서가 있다
- ☐ Offer Sheet(가격·MOQ·Incoterms)를 준비했다

### 🟦 바이어 대응
- ☐ 현재 접점이 있는 바이어가 있다
- ☐ 해외 수출 경험이 있다
- ☐ 박람회 참가 경험이 있다

**설계 포인트**
- "모르겠음/없음" 항목 삭제 — 미체크가 곧 없음/갭. 선택지가 줄어 인지부하가 낮아진다.
- 영역 상단 1줄 설명으로 규제 항목을 학습시킨다.
- 체크리스트로 안 담기는 맥락 정보(제품명·카테고리·목표국가)는 시작 화면에 최소 유지.

## 체크리스트 → 점수 매핑

기존 4영역 점수·우선과제·갭·EU현황 **출력 구조와 공식을 그대로 유지**하고, 입력만 체크리스트에서 파생시킨다. 기준은 현행 `backend/src/services/aiDiagnosis.service.js`.

| 체크 항목 | 채점 반영 |
|---|---|
| 국내 판매 중 | `제품 경쟁력` +25, 미체크 시 갭 "국내 판매 실적/레퍼런스" |
| 월 판매량·베스트셀러 | `제품 경쟁력` +10 |
| 제조 인증 보유 | `제품 경쟁력` +18, 미체크 시 갭 "인증 확보" |
| EU 필수 7요건 (각각) | `EU 규제` = 20 + (체크수/7)×70. 미체크 항목 = `euStatus.missing` + 갭 |
| PPWR 4항목 | `EU 규제`의 포장 점수(체크수 기반). 0개 체크 시 갭 "PPWR 대응" |
| 영문 카탈로그·회사소개서 | `영업 자료` 기준점. 미체크 시 −22 + 갭 "영문 카탈로그·회사소개서" |
| Offer Sheet | `영업 자료` 갭 항목. 체크 시 해소 |
| 접점 바이어 있음 | `바이어 대응` +25. 미체크 시 갭 "타겟 바이어 리스트" + 우선과제 "바이어 발굴" |
| 수출 경험 | `바이어 대응` +15 |
| 박람회 경험 | `바이어 대응` Follow-up 판단에 반영 |

- **종합점수 = 4영역 평균** (기존 동일). 준비수준 임계값(75/55) 유지.
- **우선 해결 과제 Top 3**: 기존 가중치 로직 그대로(EU 미비 개수 → 바이어 발굴/Follow-up → 영문자료 → PPWR → 인증). 체크리스트라 "미체크=갭"이 우선과제 풀로 바로 연결돼 로직이 더 단순·명확.
- 점수 공식·임계값·가중치는 기존 값 계승. 새 스택에서 TS로 옮기되, **문항↔변수 대응표를 스펙에 못 박아 회귀 테스트로 검증**.

## 결과 화면 & 전환 훅

기존 결과 화면을 그대로 계승(재설계 없음):
- 종합점수 + 준비수준 배지
- 4영역 점수 카드 + 영역별 부족 항목 칩
- 우선 해결 과제 Top 3 박스
- EU 필수요건 현황(보유 n/7 + 미비 목록)
- 맞춤 요약 + 전문가 상담 전환 CTA(훅)
- 비회원: 결과 화면에서 비밀번호 설정 → 가입(결과가 계정에 연결) / 회원: 마이페이지 저장·재열람·PDF

체크리스트라 결과의 설득력이 올라간다 — "체크 안 한 것 = 갭"이라 부족 항목이 사용자 입력과 1:1 대응돼 납득이 쉽다.

## 개인정보 동의 처리

**원칙**: 동의 대상은 연락처(개인정보)이지 준비도 체크 항목이 아니다. 체크리스트 답변은 회사의 사업 준비 상태이며 개인정보가 아니다.

**필수 동의 1줄 유지(기존 패턴 계승)** — "개인정보 수집 및 이용에 동의합니다 *" 체크박스 + 이용약관·개인정보처리방침 링크. 미체크 시 제출 차단.

**게이트 2지점** (개인정보가 실제 제출되는 곳):

| 시점 | 동의 |
|---|---|
| ① 진단 제출(체크리스트 완료 → "진단 결과 보기") | 필수 동의 체크박스. 미체크 시 제출 차단. 연락처가 여기서 저장됨. |
| ② 결과 화면 비회원 가입(비번 설정) | 별도 필수 동의. 계정 생성 = 개인정보 저장. 새 스택에선 Supabase Auth 가입 지점. |

동의 게이트는 시작 화면이 아니라 **마지막 제출 버튼 직전**에 둔다(입력 중 방해 없음, "제출=수집 시점"과 일치). 마케팅 동의·동의 이력 DB 저장은 이번 범위 아님.

## 데이터 모델 (Supabase Postgres)

마이그레이션 스펙의 `export_diagnosis_requests`에 체크리스트 응답을 담는다. 자유입력 필드 다수가 사라지고 불리언 체크 묶음으로 단순화된다.

- 기존 배열 필드 `certifications`, `euComplianceReadiness`, `packagingReadiness`(text[]) → 체크된 항목 배열로 계속 사용.
- 신규 불리언성 체크(국내판매·판매실적·영문자료·Offer Sheet·바이어·수출경험·박람회) → `jsonb` 응답 묶음 또는 개별 boolean 컬럼. (구현 플랜에서 확정)
- 유지 필드: 연락처(담당자명·회사명·이메일·전화), 제품명, 카테고리, 목표국가, 홈페이지/SNS.
- `member_id uuid` null 허용(비회원 제출), 결과화면 가입 시 서버가 원자적으로 연결(마이그레이션 스펙과 동일).

## 테스트 전략

- 기존 `test:diagnosis`(6케이스)를 **체크리스트 입력 기준으로 갱신**해 이식. 입력 구조가 바뀌므로 신규 작성에 가깝다.
- "체크 조합 → 예상 점수/갭/우선과제/euStatus" 대응을 회귀 테스트로 고정. 위 매핑표가 테스트의 근거.
- 경계 케이스: 전부 미체크(최저점·전 영역 갭), 전부 체크(최고점), EU 일부만 체크(euStatus.missing 정확성), 바이어 유/무 분기.
- 각 단계 완료 시 verification-before-completion으로 실제 동작 확인 후 완료 선언.

## 리스크 / 오픈 이슈

- **불리언 체크의 저장 형태**(개별 컬럼 vs jsonb) 확정은 구현 플랜에서. 관리자 브리핑/목록이 이 데이터를 읽으므로 함께 고려.
- 기존 회원 데이터(구 진단 결과)와의 호환 — 마이그레이션은 "새로 시작"(데이터 이전 안 함)이므로 구버전 결과 폴백은 불필요. 단 마이그레이션 완료 전 구 스택 관리자 화면과 혼용되지 않도록 P2 범위 안에서 처리.
- 목표국가·카테고리는 여전히 선택 UI(칩/드롭다운)로 남으므로, 체크리스트 "한 판" 안에 자연스럽게 배치할지 시작 화면에 분리할지 UI 배치는 구현 시 확정.
- P1(다른 에이전트 진행 중)과 독립. 본 설계는 P1 완료(스캐폴드·Auth·스키마) 위에서 P2로 구현.
